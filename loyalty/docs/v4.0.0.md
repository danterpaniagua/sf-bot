# Skill Reference: Fraud Detection — v4.0.0

> **Status:** In development — branch `v4.0.0`.  
> **Breaking change:** `fraud-points` skill is split into two independent skills.

---

## What changed from v3.0.0

v3.0.0 was a single skill covering all fraud vectors. During the 2026-06-04 investigation, a new vector was confirmed: POS terminal manipulation via franchise networks. This vector requires a distinct set of tables, a different investigation workflow, and different actor memory, making it a natural split.

v4.0.0 introduces two skills:

| Skill | File | Version | Scope |
|---|---|---|---|
| `fraud-points` | `.claude/commands/fraud-points.md` | 4.0.0 | Transfer fraud, accumulation anomaly, manual assignment exploit |
| `fraud-pos` | `.claude/commands/fraud-pos.md` | 1.0.0 | POS terminal manipulation, franchise network fraud |

### Retained from v3.0.0

- Accumulation anomaly workflow

- Fraud patterns: fan-in, automation cadence, limit breach, credential stuffing, circular transfer, account consolidation, dormant activation, profile tampering, duplicate accounts, promotion exploitation, manual insertion

### Added in v4.0.0

- **Memory system integration** (Step 0a): read `known_hubs.md`, `known_relays.md`, `actor_notes.md` before issuing any queries. Flag known actors immediately.
- **Name & Identity Flags**: DNI validity by country (AR/PY/UY/CL), name plausibility rules (anagram detection, Southern Cone context), known actor lookup.
- **Email verification signals**: disposable domain detection (yopmail.com, hilostar.com + community blocklist reference), `SmlSt.CustomerMailing` absence as secondary signal. No verification column exists in the schema — confirmed 2026-06-04.
- **Collaborator profile**: `HumanResourcesPoints` history as collaborator identity signal; distinct transfer limits (30k/40k/60k daily/weekly/monthly).
- **POS pivot note**: when cross-branch POS activity is detected on an account, document the finding and signal that `fraud-pos` should be invoked for the franchise investigation path.
- **Fraud patterns added**: consecutive DNI same name, fake identity (disposable email + invalid DNI), unverified email on hub/sender.
- **Evidence folder**: `events/YYYYMMDD_fraude_transferencias/`
- **Actions section label**: "Acciones propuestas" (not "Acciones requeridas").
- **Memory update protocol**: on investigation close, update `known_hubs.md` / `known_relays.md` / `actor_notes.md`.

## `fraud-pos` v1.0.0 — Scope (new skill)

### Purpose

Investigate point diversion via manipulation of POS terminal credentials in franchise branch networks. Triggered when transfer fraud investigation reveals an account with high-volume cross-branch POS activity, or when franchise-level anomaly is reported directly.

### Core tables (new in this skill)

- `sml.Sale` — `Id, SaleDate, BranchOfficeId, CustomerId, PlatformCode, PaymentTypeCode`
- `sml.BranchOffice` — `Id, Name, Code, FranchiseGroupId, ActivatedDate, DeactivatedDate`
- `sml.FranchiseGroup` — `Id, Name`
- `sml.FranchiseStaff` — `Id, StaffId, FranchiseGroupId, StaffRoleCode, CreatedDate, DeactivatedDate`

### Investigation workflow

1. Volume by branch — same CustomerId across many branches = hardcoded credential
2. Geographic impossibility check — same account in distant provinces within minutes
3. Franchise owner mapping — `BranchOffice → FranchiseGroup`
4. Staff investigation — shared email across multiple staff = single insider with multiple roles
5. Point yield by branch — identify primary diversion source vs. zero-point terminals
6. Redemption link — cross-reference with `DiscountPointsByExchange` to find cashout branches

### Memory integration

Reads `loyalty/memory/known_pos.md` at start. Updates it at close: accumulation branches, redemption branches, insider staff.

### Fraud patterns

- POS credential hardcoded — same CustomerId across many branches/provinces
- Multi-branch franchise network — same FranchiseGroupId across high-volume branches
- Staff identity consolidation — multiple FranchiseStaff rows sharing one email

### Evidence

- POS investigation queries: `events/YYYYMMDD_fraude_transferencias/YYYYMMDD_consultas_pdv.sql`
- Branch actor memory: `loyalty/memory/known_pos.md`
